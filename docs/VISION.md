# Brain OS — Vision & Architecture

## What it is

A portable **memory + automation layer for AI coding assistants**. Brain OS turns a plain Markdown
vault into durable, multi-project memory and gives the assistant a set of composable `/brain`
commands to read, write, and reason over it — so context survives across sessions and machines.

The product is the **method** (the skill files + conventions + installer). Your notes are *yours*:
they live in your vault, are generated by you through the commands, and are never part of what ships.

## What it achieves

- **Continuity** — the assistant resumes a project with its status, decisions, agenda, and history,
  instead of starting blind every session.
- **Correctness without discipline** — structure is *projected from the filesystem* by `/brain sync`,
  not dependent on the assistant remembering to link notes. Orphans become impossible by construction.
- **Scales by tiering, not loading** — structured recall is a direct file read (O(1) per project);
  associative recall ("what across all my work relates to X") is the optional graphify graph.
- **Portable** — one config + one installer stands up the whole brain (skills + vault skeleton) on
  any machine. No hardcoded paths (verified: a clean install leaves zero unresolved tokens).

## The architecture: Journal · Map · Hands

| Layer | Engine | Role |
|-------|--------|------|
| **Journal** | Markdown vault | the system of record: intent, decisions, status, session log, agenda |
| **Map** | graphify *(optional)* | code + notes as one queryable graph, for associative recall |
| **Hands** | agenda + loops | execute tasks against a project's prioritized backlog |

- **Two stores, one graph.** Code stays in its repo; the journal stays in the vault. `/brain map`
  stages both into one corpus so the graph links *intent to implementation* — without co-locating
  files. (Verified: a concept node like `IDM` traverses into the code that implements it.)
- **Deterministic structure.** Projects use the prefixed folder-note pattern
  (`{name}.md`, `{name}-status.md`, …). `/brain sync` enforces the set and indexes loose notes.
- **The wrapper.** `install.{sh,ps1}` resolves path tokens, deploys the skills, and bootstraps a
  fresh vault skeleton from `templates/vault/` (idempotent). `/brain init` does the same in-session.
  From `git clone` to a working brain is one command.

## Design rationale (the honest journey)

Brain OS was built by using it. The shape emerged from real failures, each of which hardened the design:

1. **Drift fix** — read and write paths disagreed on filenames (prefixed vs unprefixed), so writes
   were silently unrecalled. Lesson: *one source of truth, enforced by a checker* — now `/brain sync`.
2. **Graphify as the Map** — a hand-maintained wikilink graph always orphans and breaks; deriving the
   graph from content is the robust path. graphify became the Map layer, retiring a heuristic index.
3. **Agenda + loops (Hands)** — gave the Tier-C "background worker" idea a concrete job description:
   a prioritized backlog the assistant works top-down, with outward/irreversible actions gated.
4. **The portable wrapper** — the method is only useful if anyone can stand it up; templates + a
   bootstrapping installer make "create the whole brain architecture" a single command.

A recurring principle runs through all four: **prove behavior, don't assert structure.** Every layer
ships only after a real run shows it working — not after the files merely exist.
